Key Takeaways from the Paper: "Discovering Governing Equations from Partial Measurements with Deep Delay Autoencoders"

The paper, authored by Joseph Bakarji, Kathleen Champion, J. Nathan Kutz, and Steven L. Brunton, introduces a framework for discovering interpretable governing equations of dynamical systems from incomplete or partial measurements. It builds on Takens’ embedding theorem, which uses time-delayed measurements to reconstruct the full system's attractor from limited observations.arxiv.org

Main Contributions:
A novel deep delay autoencoder that learns a coordinate transformation from delay-embedded partial data to a latent space where the dynamics can be sparsely represented in closed form.
Integration of Sparse Identification of Nonlinear Dynamics (SINDy) with deep learning to produce interpretable models, even from a single measurement variable.
Demonstrations on classic chaotic systems (Lorenz, Rössler, Lotka-Volterra) and real-world data from a chaotic waterwheel experiment, achieving sparser models than originals in some cases (e.g., 6 terms for Lorenz instead of 7).
Methodology:
Time-delay embedding augments partial observations to create a higher-dimensional space equivalent to the full system.
A deep autoencoder maps this to a low-dimensional latent space, optimized with custom losses: reconstruction, SINDy regularization in both delay and latent spaces, and consistency constraints.
Training progresses from supervised (with known states) to fully unsupervised, using techniques like periodic SINDy optimization to avoid local minima and overfitting.
Key Results:
Successfully discovers accurate, sparse models for partially observed systems, reproducing chaotic attractors and dynamics.
For the waterwheel video, extracts angular velocity from footage and learns a Lorenz-like model using polynomial features, handling noise effectively.
Highlights challenges like initialization sensitivity, suggesting better performance with physics-informed priors.
Conclusions and Implications:
This approach addresses a major gap in data-driven modeling by enabling equation discovery from incomplete data, with applications in physics, engineering, and beyond.
Future directions include advanced architectures (e.g., convolutional networks) and improved handling of noise via probabilistic methods.
Overall, it emphasizes the power of combining delay embeddings, deep learning, and sparsity for interpretable science from limited observations.

The phrase "latent space where the dynamics can be sparsely represented" refers to a transformed, lower-dimensional coordinate system (learned via a deep autoencoder) in which the time evolution of a dynamical system can be described by a simple, closed-form differential equation that includes only a minimal number of mathematical terms. This is achieved through techniques like Sparse Identification of Nonlinear Dynamics (SINDy), which selects a "sparse" set of functions from a larger library of candidates, prioritizing parsimony (fewer terms) for better interpretability and generalization, akin to Occam's razor.arxiv.org

No, this does not mean the governing equations are limited to linear terms involving single variables (e.g., 5x + 7y). Instead, "sparsely represented" allows for nonlinear terms such as products of variables (e.g., 8xy) or higher powers (e.g., 9xz²), but only the essential ones are included, with most potential terms having zero coefficients to keep the model concise. For instance, in the paper's application to the Lorenz system, the discovered equations in the latent space feature a mix of linear terms (e.g., -16z₁ or 8z₂) and nonlinear terms (e.g., 2.5z₂z₃, -16z₁z₃, 8z₂², or -1.4z₂³), resulting in a sparser overall model (6 terms) compared to the original Lorenz equations (7 terms). The library of candidate functions often includes polynomials up to cubic order or even trigonometric terms, enabling these nonlinear interactions while enforcing sparsity.

Key MSRs for CPU Usage

CPU utilization is calculated from unhalted cycles (actual work) vs. reference cycles (maximum possible). Use these fixed-function counters (enabled via MSR 0x38D for control):

IA32_FIXED_CTR1 (0x30A): Unhalted core cycles (actual cycles the core was active).
IA32_FIXED_CTR2 (0x30B): Unhalted reference cycles (cycles at fixed frequency, for normalization).
Utilization formula: (unhalted_core_cycles_delta / unhalted_reference_cycles_delta) * 100 over your sampling interval.

For multi-core, read per-core (via CPU affinity) and average. These are faster than programmable counters like PERF_COUNT_HW_CPU_CYCLES (0xE7/0xE8 for MPERF/APERF, which are similar but less precise for utilization).

#define MSR_FIXED_CTR_CTRL 0x38D  // Enable counters (set to 0x333 for ring 0-3)

int main() {
    int fd = open("/dev/cpu/0/msr", O_RDONLY);  // For core 0; loop over cores for average
    if (fd < 0) {
        perror("open /dev/cpu/0/msr");
        return 1;

    // Enable counters if needed (requires WRONLY and wrmsr)
    // For simplicity, assume already enabled (default on many systems)

    uint64_t prev_core = 0, prev_ref = 0;
    while (1) {
        uint64_t core_cycles, ref_cycles;
        if (pread(fd, &core_cycles, sizeof(uint64_t), MSR_FIXED_CTR1) != sizeof(uint64_t)) {
            perror("pread MSR_FIXED_CTR1");
            break;
        }
        if (pread(fd, &ref_cycles, sizeof(uint64_t), MSR_FIXED_CTR2) != sizeof(uint64_t)) {
            perror("pread MSR_FIXED_CTR2");
            break;
        }

        double utilization = 0.0;
        if (ref_cycles > prev_ref) {
            utilization = 100.0 * (double)(core_cycles - prev_core) / (double)(ref_cycles - prev_ref);
        }
        printf("CPU Utilization: %.2f%%\n", utilization);

        prev_core = core_cycles;
        prev_ref = ref_cycles;
        usleep(1000);  // 1ms interval
    }

    close(fd);
    return 0;

This allows you to specify an array of MSR operations (e.g., read unhalted core/reference cycles for utilization, plus PKG_ENERGY_STATUS 0x611 for RAPL power) and execute them via a single ioctl(2) system call with X86_IOC_MSR_BATCH. The batch is processed as one kernel operation, which minimizes time differences between reads (e.g., reducing overhead from multiple system calls/context switches to microseconds or less, compared to separate reads that could drift by milliseconds due to scheduling).

It's not strictly atomic (operations are executed sequentially within the call, and errors are per-op), but the timing is much tighter than individual rdmsr or pread calls—often within the same scheduling quantum. For example, you can define a struct msr_batch_array with ops for multiple MSRs across CPUs, and the kernel handles the batch efficiently.
